01 / AttractorFlow Plugin

A — What is AttractorFlow?

AttractorFlow is a Model Context Protocol (MCP) plugin that treats autonomous agent trajectories as dynamical systems. Each agent step is encoded into a 384-dim semantic embedding using a lightweight sentence-transformer, appended to a rolling trajectory buffer.

The plugin continuously computes Finite-Time Lyapunov Exponents (FTLE) — a measure borrowed from chaos theory that quantifies how quickly nearby trajectories diverge. A positive λ indicates expanding (exploratory) dynamics; a strongly negative λ signals converging behavior toward a stable attractor.

From the FTLE time series, AttractorFlow classifies the agent into one of 7 trajectory regimes, each with a prescribed intervention hint. This allows the orchestrating agent to adaptively steer sub-agents — amplifying convergence when improvement is happening, and perturbing the trajectory when the agent is stuck, oscillating, or diverging.

The 8 MCP tools expose this machinery: Recording tools capture step state; Diagnostic tools query regime and basin geometry; Intervention tools reshape the trajectory when needed.

B — 8 MCP Tools
record_state
Recording
Embeds step text into 384-dim vector, updates trajectory buffer and recomputes FTLE.
Called: after every agent step
checkpoint
Recording
Saves a stable trajectory point as a named anchor for rollback and basin depth reference.
Called: when tests pass
get_regime
Diagnostic
Returns current regime, confidence score, and intervention_hint for the orchestrator.
Called: every 4 steps
get_lyapunov
Diagnostic
Returns SVD-based FTLE (λ = log(σ_max)/W), isotropy_ratio (σ_min/σ_max), singular_values[0..2], and step_growth_rate (legacy). New fields detect 1D trajectory collapse even when global λ looks healthy.
Called: every 3 steps in convergence phase
get_trajectory
Diagnostic
Returns embedding trajectory; 2D PCA projection for visualization only. Analysis (PITCHFORK, SADDLE_NODE) runs in full 384-dim space.
Called: on /evolve-status
get_basin_depth
Diagnostic
Estimates depth of current attractor basin before committing an improvement.
Called: before merge / commit
detect_bifurcation
Intervention
k-means clustering of proposals; computes silhouette score to detect PITCHFORK / HOPF events.
Called: every cycle
inject_perturbation
Intervention
Perturbs the trajectory embedding to escape attractor traps and resume productive exploration.
Called: on STUCK / PLATEAU / OSCILLATING
C — λ (Lyapunov) Zone Map
D — 7 Trajectory Regimes
EXPLORING
Agent actively scanning quality axes; λ ∈ [0.05, 0.25]. Healthy divergence.
CONVERGING
Agent implementing improvements toward attractor; λ < −0.05. Desired state.
STUCK
Agent repeating identical proposals. Low embedding variance. Needs perturbation.
OSCILLATING
Agent alternates between two states (e.g. Union[str,None] ↔ Optional[str]). High autocorrelation.
DIVERGING
Agent rewriting whole modules instead of targeted fixes; λ > 0.25. Scope escalation.
PLATEAU
Tiny incremental reductions (cc: 8.0→7.9→7.85). Marginal returns. Needs new axis.
CYCLING
Healthy TDD loop: write test → pass → refactor. Productive periodic orbit.

Attractor Theory & Dynamics Engine

How AttractorFlow models agent trajectories as physics

A — The Embedding Pipeline

sentence-transformers all-MiniLM-L6-v2  ·  384-dim L2-normalized  ·  normal step distance: 0.5–1.4  ·  stuck threshold: < 0.40

B — The FTLE Formula (SVD-based, 384-dim)

M = displacement matrix  (W × 384)
λ = log(σmax(M)) / W
W = 8  (sliding window)  ·  σmax = largest singular value
Mij = embeddingi[j] − embeddingi−1[j]  (rows = steps, cols = dims)
isotropy = σmin/σmax  ·  1.0 = full, ~0 = 1D collapse
λ < −0.2 CONVERGING strongly
−0.2 to −0.05 CONVERGING weakly
−0.05 to 0.05 NEUTRAL
0.05 to 0.25 EXPLORING
λ > 0.25 DIVERGING
Converging Example
d₁=0.82, d₂=1.10, d₃=0.65, d₄=0.43
ln(1.10/0.82) = +0.29
ln(0.65/1.10) = −0.52
ln(0.43/0.65) = −0.41
mean = −0.64 / 3 ≈ −0.21
→ CONVERGING weakly (λ = −0.21)
Exploring Example
d₁=0.55, d₂=0.72, d₃=0.88, d₄=1.05
ln(0.72/0.55) = +0.27
ln(0.88/0.72) = +0.20
ln(1.05/0.88) = +0.18
mean = +0.65 / 3 ≈ +0.22
→ EXPLORING (λ = +0.22)

E — PITCHFORK Detection (k-means in 384-dim Embedding Space)

k-means (k=2) directly in 384-dim space  ·  Silhouette score = 0.72 ≥ 0.60 threshold  →  PITCHFORK fires  ·  PCA 2D used for centroid visualization only

F — Bifurcation Types (from bifurcation.py)

PITCHFORK
Proposal space splits into two distinct semantic clusters
X = embeddings_matrix     # W × 384
labels = KMeans(k=2).fit(X)
score = silhouette_score(X, labels)
if score ≥ 0.60: PITCHFORK
# PCA(2) only for centroid viz
Response: spawn 2 parallel convergence-agents — one per cluster centroid
HOPF
CONVERGING phase transitions to cyclic orbit
prev_λ = lyapunov(t−1)
curr_λ = lyapunov(t)
if prev_λ < 0 and curr_λ > 0
   and regime == CYCLING: HOPF
Response: add explicit exit criterion — "stop when cc < 5"
SADDLE_NODE
High condition number κ signals near-singular trajectory collapse
M = displacement matrix (W×384)
σ = singular_values(M)
κ = σ_max / σ_min
if κ ≥ 100: SADDLE_NODE
# κ→∞ = all steps in 1 direction

AttractorFlow vs Baseline

Same task · same target · same 5 cycles · same codebase.
One system has AttractorFlow. The other does not.

Methodology
Both runs started from an identical weakened baseline (coverage=41.2%, mypy=51, bench=17ms, cc=3.07) via scripts/reset_baseline.sh. Runs executed sequentially on 2026-03-28 — greedy first, then AF on a fresh reset. Baseline used a greedy sequential agent with no MCP tools — picks improvements by inspection, always serial, no stuck/oscillation detection, no parallel tracks, no checkpointing. AttractorFlow monitored the trajectory via SVD-FTLE, called attractorflow_detect_bifurcation() and attractorflow_get_regime() each cycle. All 5 cycles converged cleanly (regime=CONVERGING, FTLE < 0 throughout) — no PITCHFORK bifurcation or perturbation injections were needed.
